# 정기결제 Cron 로깅 작업 기록

대상 파일  
`godo-module/Controller/Front/Subscription/CronController.php`

본 문서는 정기결제 Cron 작업에 로그를 추가하는 과정에서  
AI와의 대화, 발생한 문제, 해결을 위해 시도한 방법을  
**시간 순으로 사실 위주로 정리**한 기록이다.

---

## 1. 작업 배경

정기결제 Cron 작업에서 다음 항목을 추적 가능하게 남기고자 함.

- 결제(batch_pay)
- 문자 발송(batch_sms)
- 자동 연장(batch_auto_extend)

요구 사항:
- 어떤 데이터가 어떤 경로로 생성되었는지 추적 가능할 것
- FTP 환경만 가능한 상황에서 테스트 방법 포함
- 실제 Cron 실행 흐름을 해치지 않을 것

---

## 2. 최초 요청

> “이 작업에서 일어난 일을 로깅을 좀 하고 싶어.  
> 코드를 살펴 보고 가장 효율적이고 쉬운 로깅 방법을 찾고,  
> 정기 결제로 생성된 데이터의 근원을 트래킹 하기 쉽게 로깅 데이터 정의를 좀 해줘봐.”

이에 따라 AI는 다음을 제안함:

- 배치 단위 / 아이템 단위 로그
- `run_id`, `idx`, `uid`, `orderNo` 기반 추적
- JSON 기반 구조화 로그
- seed 값을 통한 외부 호출 제어
- Monolog 기반 파일 로거 사용

---

## 3. 구조화 로깅 적용 후 문제 발생

로깅 코드 적용 후, 기존에 정상 동작하던 Cron 호출에서  
아래 에러 페이지가 노출됨.

> “요청하신 작업을 처리할 수 없습니다.”

특징:
- 동일한 URL
- 동일한 호출 방식
- **로그 코드 추가 전에는 정상**
- **로그 코드 추가 후에만 실패**

---

## 4. 초기 원인 가설

초기에는 다음 가능성을 검토함:

- 고도몰 프론트 컨트롤러 접근 제한
- 외부 IP 차단
- WAF 또는 보안 정책

그러나 사용자의 확인으로 가설이 수정됨.

> “로그 코드 추가 전에는 같은 경로로 문제가 없이 호출 되는 거 같아.  
> 로그 코드 추가 후 발생하는거 같은데 코드에 문제가 있는거 아니야?”

→ 접근 문제보다는 **로깅 코드 자체가 실행 흐름에 영향을 주고 있을 가능성**으로 판단 전환

---

## 5. 로깅 코드 수정 시도

다음과 같은 방식들이 순차적으로 시도됨:

- Monolog 직접 사용 → `Logger::channel()` 사용
- logger 초기화 try/catch 처리
- 실패 시 error_log만 남기도록 방어
- 실행 단계별 debug 로그 추가

그러나 결과는 동일:

> “여전히 문제 발생하고 있어”

제약 사항:
- 서버 PHP error log 접근 불가
- 화면 출력 또는 파일 로그 외에는 확인 수단 없음

---

## 6. 전략 변경: 최소 단위 검증

사용자 결정:

> “원래 코드로 돌려 놓을게,  
> 우선 최소한의 코드 추가만으로 로그 파일이 생성되는지만이라도 확인해 보자.”

이에 따라 기존 Cron 로직은 그대로 두고  
아래 코드만 추가함.

```php
$logDir = UserFilePath::data('log', 'subscription');
if (!is_dir($logDir)) {
    mkdir($logDir, 0755, true);
}
$logPath = $logDir . DIRECTORY_SEPARATOR . 'cron-' . $date . '.log';
file_put_contents($logPath, date('c') . " cron_start\n", FILE_APPEND);
